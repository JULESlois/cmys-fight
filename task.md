# CMYS Fight 第一优先级工作：战斗—路线垂直切片重构

## 一、首要目标

建立一段完整、可重复游玩的流程：

```text
主城配置
→ 进入一个章节
→ 使用两种不同资源模型的武器战斗
→ 获得会改变操作循环的 Buff
→ 发现一个隐藏房
→ 在两个出口中选择下一章节
→ 路线选择影响后续敌人、场景和奖励
```

这个垂直切片必须同时验证：

* 新武器资源模型是否成立。
* 双武器切换是否有实际意义。
* 通用闪避与角色技能是否能够共存。
* Buff 是否改变动作，而不是只增加数值。
* 场景建筑是否能够参与战斗。
* 隐藏房是否具备发现过程。
* 多出口章节是否能够改变远征路线。
* 主城建筑是否能够配置并记录这些系统。

在该切片成立之前，不应全面迁移57把武器，也不应批量增加章节和 Buff。

---

# 二、为什么这是最高优先级

目前项目的场景表现和建筑架构已经明显超过战斗底层：

* 主城已经具备实体建筑、碰撞、遮挡和交互区域。
* 地牢已经具备主题结构、可破坏物和实体设施。
* 房间生成、门、传送门和小地图已经存在。
* 双武器槽、角色技能、Buff选择也已经存在。

但这些系统仍然主要通过旧规则连接：

* 多数武器共享同一个能量池。
* 武器特殊状态集中保存在玩家对象上。
* Buff主要是硬编码数值查询。
* 技能持续时间与冷却规则不统一。
* 场景设施多为一次性免费奖励。
* 出口仍然只是“进入下一固定关卡”。
* 隐藏房和路线选择缺少正式的数据结构。

继续在旧结构上增加内容，会让后续重构成本迅速增加。

因此第一阶段要做的不是“内容扩充”，而是：

> 建立能够承载未来内容的最小正式架构，并用一段真实流程证明它可玩。

# 四、第一阶段：重构远征与章节数据

这是多出口章节和隐藏路线的前置条件。

## 4.1 分离章节身份与难度深度

当前不能继续让 `chapterIndex` 同时决定：

* 章节内容。
* 场景主题。
* 难度。
* 线性推进顺序。

改为：

```ts
interface RunProgress {
  worldNodeId: string;
  routeDepth: number;
  stageWithinNode: number;
  routeHistory: string[];

  hardMode: boolean;
  challengeId?: string;
}
```

职责：

* `worldNodeId`：当前区域，例如 `overgrown_archive`。
* `routeDepth`：本局推进深度，控制敌人数值和奖励品质。
* `stageWithinNode`：当前区域内部进度。
* `routeHistory`：记录玩家选择过的章节。

核心原则：

> 章节决定内容，路线深度决定强度。

## 4.2 建立世界节点定义

```ts
interface WorldNodeDefinition {
  id: string;
  name: string;
  theme: EnemyTheme;

  enemyTags: string[];
  hazardTags: string[];
  rewardBiases: string[];

  exits: WorldExitDefinition[];
}
```

第一版只制作三个节点：

```text
overgrown_archive
cooling_canal
sealed_armory
```

路径：

```text
蔓生档案馆
├─ 冷却运河
└─ 封印军械库
```

先验证一次真正的路线分支，不制作完整世界图。

## 4.3 改造出口房

```ts
interface ExitDestination {
  worldNodeId: string;
  kind:
    | "normal"
    | "hidden"
    | "performance";

  state:
    | "hidden"
    | "locked"
    | "available"
    | "chosen";

  preview: {
    threat: number;
    enemyTags: string[];
    hazardTags: string[];
    rewardTags: string[];
  };
}
```

进入出口后才提交路线选择。

进入出口房查看信息不等于立刻离开当前章节。

# 五、第二阶段：建立正式武器运行时

这是整个战斗重构的核心。

## 5.1 从玩家对象中移出武器状态

当前这些状态不应继续全部放在 `Player`：

* 热量。
* 连发阶段。
* 蓄力时间。
* 底火/引爆状态。
* 悠悠球状态。
* 武器专属冷却。

改为每个武器槽保存独立状态：

```ts
interface WeaponRuntimeState {
  weaponId: string;
  resourceType:
    | "magazine"
    | "battery"
    | "heat"
    | "charge"
    | "action";

  fireCooldown: number;
  resourceState: WeaponResourceRuntime;
  customState: Record<string, number | boolean | string>;
}
```

玩家只保存：

```ts
interface WeaponLoadoutRuntime {
  slots: [
    WeaponRuntimeState,
    WeaponRuntimeState?
  ];

  activeSlot: 0 | 1;
  swapTimer: number;
}
```

### 弹匣

验证武器：

功能：

* 打空弹匣事件。

### 电池

验证武器：

功能：

* 收起时加速恢复。

### 热量

验证武器：

功能：

* 停火与收起冷却。

### 充能槽

# 六、第三阶段：双武器战斗循环

双武器必须成为核心循环，而不是额外按钮。

## 6.1 基础规则

* 每把武器独立保存资源状态。
* 非当前武器可以后台换弹、冷却或恢复。
* 换枪具有极短准备时间。
* 换枪不能无条件取消所有后摇。
* 重型武器准备时间略长。

建议初始值：

| 类型 | 换出准备时间 |
| -- | -----: |
| 轻型 |  0.15秒 |
| 普通 |  0.22秒 |
| 重型 |  0.32秒 |

## 6.2 垂直切片应验证三种组合

### 弹匣＋弹匣

一把换弹时切换另一把。

### 电池＋弹匣

电池武器收起恢复，弹匣武器维持输出。

### 热量＋充能

热量武器冷却期间，使用重型充能攻击。

## 6.3 合格标准

玩家应当主动换枪，而不是因为当前武器完全无法使用才被迫换枪。

目标不是无限持续输出，而是形成：

```text
爆发
→ 资源进入停顿
→ 换枪
→ 另一把武器承担战斗
→ 原武器后台恢复
→ 再次切回
```

# 七、第四阶段：统一闪避和技能生命周期

## 7.1 通用战术闪避

所有角色获得基础闪避：

* 冷却约0.8秒。
* 无敌时间约0.17秒。
* 位移距离根据角色变化。
* 闪避期间不能正常攻击。
* 支持完美闪避事件。

角色差异：

### Knight

* 位移较短。
* 完美闪避推进护甲恢复。

### Mage

* 位移中等。
* 完美闪避增加奥术回响。

### Rogue

* 位移最长。
* 闪避可以取消部分换枪准备。
* 完美闪避获得短暂暴击窗口。

Rogue现有角色技能不能与通用闪避完全重复，需要重新定义成更强的进攻型影袭或连续位移。

## 7.2 技能生命周期

技能定义增加：

```ts
interface SkillDefinition {
  cooldown: number;
  duration: number;

  cooldownStart:
    | "on_activate"
    | "on_effect_end";

  refreshPolicy:
    | "disabled_while_active"
    | "refresh_duration"
    | "replace_instance";
}
```

规则：

* 瞬发技能可以从激活时开始冷却。
* 长持续技能原则上从效果结束后开始冷却。
* 持续技能生效期间不能无条件重新激活。
* 通用技能冷却缩减上限先设为15%。
* 不再允许多个百分比乘算将冷却压缩40%。

第一版只完整处理：

* Knight。
* Mage。
* Rogue。

# 八、第五阶段：建立战斗事件层

这是新 Buff、角色被动、武器模块和场景互动的共同基础。

```ts
type CombatEvent =
  | "weapon_fired"
  | "weapon_swapped"
  | "magazine_empty"
  | "reload_started"
  | "reload_completed"
  | "battery_depleted"
  | "battery_full"
  | "weapon_overheated"
  | "charge_restored"
  | "dodge_started"
  | "perfect_dodge"
  | "skill_activated"
  | "armor_broken"
  | "enemy_killed"
  | "prop_destroyed"
  | "hazard_kill"
  | "room_perfect_clear"
  | "boss_perfect_clear"
  | "facility_used"
  | "exit_inspected";
```

## 8.1 事件执行顺序

必须明确：

1. 基础动作发生。
2. 武器资源消费。
3. 角色被动处理。
4. 武器模块处理。
5. Buff处理。
6. 协同处理。
7. 视觉与音效反馈。
8. 状态持久化。

避免不同系统直接互相修改导致递归触发。

## 8.2 第一版禁止的实现方式

不要继续大量增加：

```ts
if (BuffSystem.has(player, "..."))
```

尤其不能继续散落到：

* `DungeonState`
* `WeaponController`
* `SkillController`
* `DamageSystem`

这些类应发送事件，由 Buff 或模块处理器响应。

# 九、第六阶段：重做第一批 Buff

第一版只制作十二个 Buff，不追求数量。

## 9.1 全局战斗 Buff

### 1. 反相反击

完美闪避后，下一次攻击获得强化。

### 2. 快速换装

换枪准备时间缩短。

### 3. 壁垒记忆

靠近大型场景建筑时加速护甲恢复。

### 4. 危险解析镜

敌人受到的环境伤害提高，玩家环境伤害降低。

## 9.2 武器资源 Buff

### 5. 战术装填

切换武器后，收起武器的后台换弹速度提高。

### 6. 最后一发协议

弹匣最后一发获得特殊效果。

### 7. 满载电池

电池自然恢复至满时强化下一次攻击。

### 8. 深度放电

电池低于一定比例时增加穿透，但耗尽后的恢复延迟增加。

### 9. 热区稳定器

高热量提高攻击表现，但过热惩罚延长。

### 10. 紧急排热

完美闪避降低当前武器热量。

### 11. 充能回路

无伤清理战斗房后恢复一个充能槽。

### 12. 重型预备

切换至充能武器时，其第一发准备速度提高。

## 9.3 第一批命名协同

只实现四个：

* 冷热交换。
* 最后一发协议。
* 相位换装。
* 壁垒回路。

每个协同必须：

* 有明确名称。
* 有独立图标或提示。
* 能够直接观察到。
* 不只是两个数值相加。

# 十、第七阶段：场景与隐藏房验证

## 10.1 一个活性场景房

在蔓生档案馆中制作一个场景互动房：

* 可破坏孢子囊。
* 可燃藤蔓。
* 一处大型可作为掩体的结构。
* 一面带提示的裂纹墙。
* 一条普通绕行路径。

允许：

* 火焰快速点燃。
* 爆炸快速破坏。
* 普通攻击缓慢打开。
* 近战安全清理。

不能让某种武器成为唯一解法。

## 10.2 一个隐藏房

隐藏房类型：

**协同密室**

入口提示同时包含：

* 墙面视觉裂纹。
* 靠近后的音效变化。
* 清理房间后的短暂灰尘粒子。

隐藏房奖励：

* 根据当前武器资源模型提供三个相关模块。
* 不直接随机发放纯金币。
* 不保证最优协同，只提高相关性。

## 10.3 一个挑战裂隙

第一版可选，优先级低于普通隐藏房。

若实现：

* 入口为隐藏符号。
* 奖励提前展示。
* 内容为短弹幕或环境挑战。
* 完成后返回原房间。
* 不增加章节推进深度。

## 出口A：冷却运河

* 闪避与电池模块倾向。
* 恢复资源较稳定。

## 出口B：封印军械库

显示：

* 重甲与精英敌人。
* 弹匣和充能模块倾向。
* 武器奖励更高。

两个出口的路程成本应接近。

* 查看两个出口。
* 返回商店或隐藏房。

激活后：

* 记录目的地。
* 另一出口标记为跳过。
* 路线不可反悔。

# 十三、测试和数据采集

这个阶段不能只看理论DPS。

## 13.1 单武器指标

* 首次资源阻塞时间。
* 10秒攻击次数。
* 30秒攻击次数。
* 资源停顿占比。
* 后台恢复贡献。
* 换弹或过热次数。
* 实际直接伤害。
* 对敌群和单体的有效表现。

## 13.2 双武器指标

* 每分钟主动换枪次数。
* 两把武器使用时间占比。
* 资源完全闲置时间。
* 两把武器同时不可用的时间。
* 后台恢复产生的有效收益。
* 是否存在一把武器长期不使用。

## 13.3 Buff指标

* Buff被选择率。
* Buff触发次数。
* 每次触发平均收益。
* 是否只对极少数武器有效。
* 是否导致某类武器取消资源限制。
* 是否产生可感知的操作变化。

## 13.4 路线指标

* 两个出口选择率。
* 玩家检查两个出口的比例。
* 隐藏房发现率。
* 隐藏房奖励接受率。
* 当前Build与出口选择的关联。
* 是否存在明显的固定最优路线。

# 十六、完成标准

只有满足以下条件，这个首要里程碑才算完成：

1. 三名角色均拥有通用闪避和有效角色差异。
2. 八把测试武器使用至少四种资源模型。
3. 两把武器拥有独立资源状态并能后台恢复。
4. 玩家主动换枪能提高战斗效率。
5. 长持续技能不会通过冷却缩减永久覆盖。
6. Buff通过事件触发，不继续大量散落硬编码。
7. 至少十二个Buff能够改变具体动作。
8. 至少四个命名协同具有明显反馈。
9. 一个场景房能够通过多种方式互动。
10. 一个隐藏房具有视觉、声音或动态提示。
11. 章节末存在两个通往不同区域的出口。
12. 两个出口具有不同敌人、环境与奖励倾向。
13. 路线深度与章节身份完全分离。
14. 主城能够配置起始构筑并记录路线发现。
15. 旧存档能够迁移或被明确、安全地重置。
16. 相关自动测试和运行时指标全部通过。

# 十七、最终优先级结论

当前最高优先级可以压缩为一句话：

> **先用三名角色、八把武器、十二个Buff、一个隐藏房和一次双出口选择，完成新战斗架构的垂直切片。**

其中真正的底层优先顺序是：

```text
整理当前工作区
→ 路线数据模型
→ 武器独立运行时
→ 武器资源策略
→ 闪避与技能生命周期
→ 战斗事件层
→ 第一批Buff
→ 场景隐藏房
→ 双出口章节
→ 平衡验证
```

不要先从Buff文案或新章节美术开始。

最先应落地的代码是：

1. `RunProgress`的路线身份与难度分离。
2. `WeaponRuntimeState`及双槽独立状态。
3. 武器资源策略接口。
4. 技能生命周期规则。
5. 战斗事件分发层。

这五项完成后，后续Buff、隐藏房、场景建筑和多出口章节才不会继续建立在原型级耦合结构上。
